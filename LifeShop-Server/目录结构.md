```
┌─────────────────────────────────────┐
│      前端 (hmdp-vue3)               │  ← 唯一的客户端
└──────────────────┬──────────────────┘
                   │ HTTP 请求
┌──────────────────▼──────────────────┐
│   hmdp-core-service (核心服务)      │  ← 唯一的后端服务
│  （实际业务逻辑的主应用）            │
└──────────────────┬──────────────────┘
         ▲         │         ▲
         │ 依赖    │ 依赖    │ 依赖
    ┌────┴─────────┼─────────┴─────┐
    │              │                │
    ▼              ▼                ▼
┌─────────┐  ┌──────────┐    ┌──────────────┐
│基础工具 │  │Redis工具 │    │其他框架组件  │
│hmdp-    │  │hmdp-    │    │hmdp-        │
│common   │  │redis-   │    │sharding     │
│hmdp-    │  │tool-    │    │hmdp-        │

└─────────┘  └──────────┘    └──────────────┘
```

## 📦 各模块详解

### **1. hmdp-common（通用基础模块）**
```java
用途：项目通用工具和基础配置
包含内容：
  - 工具类（编解码、JSON 处理等）
  - JWT token 处理
  - 日志配置（Log4j2）
  - 依赖库：commons-codec, hutool, fastjson
```
**为什么独立？** 让其他所有模块都能复用基础功能，避免重复代码。

### **2. hmdp-parameter（参数管理）**
```java
用途：统一管理系统参数和验证规则
包含内容：
  - 参数 POJO
  - 参数验证
  - 参数转换工具
```

### **3. hmdp-sharding（分库分表组件）**
```java
用途：数据库分片处理
依赖：ShardingSphere JDBC（Apache 分片中间件）
为什么独立？ 
  - 分库分表是独立的专业领域
  - 可以被 hmdp-core-service 引入使用
  - 保持架构清晰，职责单一
```

### **4. hmdp-id-generator-framework（分布式ID生成）**
```java
用途：生成全局唯一的分布式 ID
基于：百度开源代码改造
特点：
  - 支持数据库方式
  - 支持 Redis 方式（自动生成 workId）
```

### **5. hmdp-mq-framework（消息队列框架）**
```
分为三层（遵循生产者-消费者模式）：
├── hmdp-mq-common-framework/     # 公共定义
├── hmdp-mq-producer-framework/   # 生产者（发送消息）
└── hmdp-mq-consumer-framework/   # 消费者（消费消息）
使用：Kafka 中间件
```

**为什么这样设计？**
- **生产者和消费者分离**：不同的应用可能只需要其中一种
- **公共层统一管理**：Topic、序列化、异常处理等共通逻辑
- **便于独立部署**：消费者可以单独横向扩展

### **6. hmdp-redis-tool-framework（Redis工具框架）**
```
分为两层：
├── hmdp-redis-common-framework/  # Redis 基础配置、连接
└── hmdp-redis-framework/         # Redis 业务方法封装
```

**为什么分层？**
- `redis-common`：管理底层连接、配置
- `redis-framework`：提供业务级别 API（如缓存、分布式锁等）

### **7. hmdp-redisson-framework（Redisson 分布式锁）**
```java
用途：提供更高级的分布式锁实现
基于：Redisson（Redis 的高级 Java 客户端）
比 Redis 直接操作的优势：
  - 自动处理过期、重入、死锁
  - 提供看门狗（watchdog）机制
  - 支持公平锁、读写锁等
```

### **8. hmdp-core-service（核心业务服务）**
```java
用途：实际的业务逻辑服务（唯一的可运行应用）
依赖：所有上面的模块
包含：
  - Controllers（API 端点）
  - Services（业务逻辑）
  - Mappers（数据库访问）
  - Models（业务模型）
```

### **9. hmdp-vue3（前端客户端）**
```javascript
技术栈：Vue 3 + Vite
用途：用户交互的前端界面
特点：
  - 集成 json-server 用于开发时 Mock 数据
  - 提供完整的开发、构建、打包支持
```

### **10. sql/（数据库脚本）**
```sql
用途：数据库初始化脚本
包含：表结构定义、初始数据等
```

## 🎯 为什么要这样设置存放内容？

### **1. 模块化与复用性 ✅**
- 每个框架都可以独立发布、版本管理
- 其他项目可以只引入需要的模块（如只引 Redis 框架）

### **2. 职责分离（SRP）✅**
```
hmdp-common         → 通用基础
hmdp-sharding       → 数据库分片
hmdp-id-generator   → ID 生成
hmdp-mq-framework   → 消息队列
hmdp-redis-*        → Redis 相关
hmdp-core-service   → 业务逻辑
hmdp-vue3           → 用户界面
```
每个模块职责单一，易于维护和测试。

### **3. 依赖隔离 ✅**
```
core-service 依赖多个模块，但各模块独立
这样如果要修改 Redis 逻辑，只需修改 hmdp-redis-* 模块
不会影响其他模块
```

### **4. 便于团队协作 ✅**
- 不同团队可以并行开发不同模块
- 清晰的依赖关系，避免循环依赖

### **5. 生产级别架构 ✅**
```
这是符合微服务思想的单体应用设计：
- 内部模块化（模拟微服务的模块独立性）
- 便于未来的微服务演进（可以逐个模块拆分为微服务）
- 目前运行为单体应用（只有一个 hmdp-core-service 在运行）
```

## 💡 核心设计理念总结

| 设计方面       | 理念                    | 好处                   |
| -------------- | ----------------------- | ---------------------- |
| **前后端分离** | Vue3 + REST API         | 独立部署、易于版本管理 |
| **框架分层**   | 通用 → 框架 → 业务      | 复用性强、降低耦合     |
| **模块独立**   | 每个模块都是 Maven 模块 | 版本管理、依赖清晰     |
| **单一职责**   | 专业的事交给专业的模块  | 代码质量高、维护容易   |
| **可测试性**   | 各模块独立测试          | 质量保证、问题快速定位 |

这是一个**工程化很强的实战项目**，非常适合学习生产级别的 Java 项目组织方式！
